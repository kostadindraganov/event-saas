# M3.3 — GDPR (Право на заличаване + Export) и Rate Limiting — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Дай на потребителя self-serve право на заличаване (GDPR анонимизация) + машинно-четим export на данните му, и защити мутациите/auth от злоупотреба с rate limiting.

**Architecture:** Заличаването е **необратима анонимизация в една транзакция** — PII на `user` реда се изтрива на място (редът остава, за да резолвират authorship join-овете към «Изтрит потребител»), обявите минават в `removed`, session/account редовете се трият. Polar customer се анонимизира best-effort след commit. Export-ът е синхронен JSON route handler. Rate limiting = Better Auth вграден (`/api/auth/*`, per-IP) + собствен in-memory sliding-window tRPC middleware за мутации (per-userId).

**Tech Stack:** Next.js 16.2.10 (forked, App Router), Better Auth 1.6.23, tRPC v11, Drizzle + neon-serverless, Polar `@polar-sh/sdk` 0.47.1, next-intl, zod v4.

## Global Constraints

- **Forked Next.js 16.2.10** — прочети релевантния guide в `node_modules/next/dist/docs/` ПРЕДИ да пишеш Next API код; `revalidateTag(tag, { expire: 0 })` е двуаргументен.
- **Само 3 tRPC procedure нива** (`src/trpc/init.ts`): `publicProcedure` / `protectedProcedure` / `adminProcedure`. Owner-мутации = `protectedProcedure` + ключ по `ctx.user.id`. НЯМА `vendorProcedure`.
- **DAL конвенция:** хвърляй `TRPCError` директно (`NOT_FOUND`/`FORBIDDEN`/`CONFLICT`/`BAD_REQUEST` + SCREAMING_SNAKE message); чужд/невалиден ресурс → `NOT_FOUND` без изброяване. CAS pattern: статус в `WHERE` + `.returning()` + `if (!row) throw CONFLICT`. Side-effects (Polar/email/revalidate) СЛЕД commit чрез `void fn().catch(console.error)` — **никога не хвърлят**.
- **next-intl:** `bg` (no-prefix) + `en` (byte-mirrored). Всеки видим низ съществува в **и `messages/bg.json` и `messages/en.json`** със същия ключов път.
- **Домейн (CONTEXT.md):** «Изтрит (анонимизиран) акаунт» е **необратим** (различен от обратимото «Блокиран акаунт»). Каноничното скрап-име е точно `Изтрит потребител`. Съдържанието на ревюта/въпроси/съобщения ОСТАВА (легитимен запис на насрещната страна); само авторството се показва анонимно. Обявите на вендора изчезват публично. Заличаване НЕ се допуска при потвърдени предстоящи резервации.
- **Секрети само в `.env`.** Нов ред в код → тест преди имплементация (TDD).

**Ключови факти от research (`.superpowers/sdd/m3.3-plan-draft/research-*.md`):**
- `listingStatus` enum вече има `"removed"` (`src/db/schema/catalog.ts:15-17`) → обявите се soft-remove-ват, което пази ревюта/резервации на насрещната страна (без cascade).
- Polar: **един** call `polarClient.customers.deleteExternal({ externalId: userId, anonymize: true })` отменя абонаментите + revoke-ва benefits + анонимизира PII при Polar. Мапинг user→Polar = `externalId = user.id` (няма локален `polarCustomerId`).
- `getCurrentUser` (`src/data/users/require-user.ts:18`) вече блокира при `deletedAt != null` — анонимизацията сетва `deletedAt` СЪЩО, така че **не е нужен нов guard там**. `anonymizedAt` служи като маркер за необратимост (блокира unblock) + audit timestamp.
- Better Auth вградените special rules вече лимитират `/sign-in`,`/sign-up`,`/change-password`,`/change-email` на **3 req/10s** (по-строго от целевото 10/min) — само `rateLimit: { enabled: true }` е достатъчно.
- tRPC мутациите се лимитират **per-userId** (`ctx.user.id` е наличен в `protectedProcedure`) → не се налага plumbing на IP в `TRPCContext`.

## Task 2: `AccountDAL.eraseAccount` — транзакцията на заличаване (CRUX)

**Files:**
- Create: `src/data/account/account.dal.ts`
- Modify: `src/lib/auth.ts` (export на `polarClient` + `hasPolar`)

**Interfaces:**
- Consumes: `user`, `booking`, `listing`, `message`, `session`, `account` schemas; `polarClient`/`hasPolar` от `@/lib/auth`.
- Produces: `AccountDAL.eraseAccount(userId: string): Promise<void>` — хвърля `CONFLICT HAS_FUTURE_BOOKINGS` / `CONFLICT ALREADY_ANONYMIZED`.

**Поведение (един tx, в този ред):**
1. Pre-check: потвърдена предстояща резервация (като клиент ИЛИ като вендор през `listing.ownerId`), `status='confirmed' AND eventDate >= днес` → `CONFLICT HAS_FUTURE_BOOKINGS`.
2. Scrub `user` реда CAS (`WHERE id AND anonymizedAt IS NULL` + `.returning()`; празно → `CONFLICT ALREADY_ANONYMIZED`): `name='Изтрит потребител'`, `email='deleted+{id}@deleted.local'`, `phone=null`, `image=null`, `emailVerified=false`, `deletedAt=now`, `anonymizedAt=now`.
3. Auto-cancel pending: като клиент → `cancelled_by_customer`; на неговите обяви → `cancelled_by_vendor` (`cancelReason='account_deleted'`).
4. Scrub `booking.phone=''` където `customerId=userId` (колоната е NOT NULL).
5. Scrub `message.phone=null` където `senderId=userId`.
6. Обяви `listing.status='removed'` където `ownerId=userId`.
7. Изтрий `session` + `account` редовете на userId.
8. СЛЕД commit (best-effort, не хвърля): `polarClient.customers.deleteExternal({ externalId: userId, anonymize: true })`.

> **Решение (потвърди при одобрение):** обявите минават в `removed` (soft), НЕ hard-delete — така ревютата/резервациите/въпросите на насрещната страна оцеляват (принцип «защита на насрещната страна»). Cloudflare Images на тези обяви **не** се трият масово (редовете оцеляват; венюто-снимки не са PII на заличаващия се) — YAGNI, отделен cron sweep ако някога дотрябва.

```ts
import { and, eq, gte, inArray, isNull, or } from "drizzle-orm";
import { TRPCError } from "@trpc/server";
import { db } from "@/db";
import { user, session, account } from "@/db/schema/auth";
import { booking } from "@/db/schema/booking";
import { listing } from "@/db/schema/catalog";
import { message } from "@/db/schema/messaging";
import { polarClient, hasPolar } from "@/lib/auth";

export class AccountDAL {
  static async eraseAccount(userId: string): Promise<void> {
    const today = new Date().toISOString().slice(0, 10);

    await db.transaction(async (tx) => {
      // 1) блокирай при потвърдена предстояща резервация (клиент ИЛИ вендор)
      const future = await tx
        .select({ id: booking.id })
        .from(booking)
        .leftJoin(listing, eq(booking.listingId, listing.id))
        .where(
          and(
            eq(booking.status, "confirmed"),
            gte(booking.eventDate, today),
            or(eq(booking.customerId, userId), eq(listing.ownerId, userId)),
          ),
        )
        .limit(1);
      if (future.length) throw new TRPCError({ code: "CONFLICT", message: "HAS_FUTURE_BOOKINGS" });

      // 2) scrub user (CAS: идемпотентност)
      const now = new Date();
      const [scrubbed] = await tx
        .update(user)
        .set({
          name: "Изтрит потребител",
          email: `deleted+${userId}@deleted.local`,
          phone: null,
          image: null,
          emailVerified: false,
          deletedAt: now,
          anonymizedAt: now,
          updatedAt: now,
        })
        .where(and(eq(user.id, userId), isNull(user.anonymizedAt)))
        .returning({ id: user.id });
      if (!scrubbed) throw new TRPCError({ code: "CONFLICT", message: "ALREADY_ANONYMIZED" });

      // 3) авто-отмени pending
      await tx
        .update(booking)
        .set({ status: "cancelled_by_customer", cancelReason: "account_deleted" })
        .where(and(eq(booking.customerId, userId), eq(booking.status, "pending")));
      await tx
        .update(booking)
        .set({ status: "cancelled_by_vendor", cancelReason: "account_deleted" })
        .where(
          and(
            inArray(booking.listingId, tx.select({ id: listing.id }).from(listing).where(eq(listing.ownerId, userId))),
            eq(booking.status, "pending"),
          ),
        );

      // 4-5) scrub контактни данни в booking/message
      await tx.update(booking).set({ phone: "" }).where(eq(booking.customerId, userId));
      await tx.update(message).set({ phone: null }).where(eq(message.senderId, userId));

      // 6) обявите изчезват публично
      await tx.update(listing).set({ status: "removed", updatedAt: now }).where(eq(listing.ownerId, userId));

      // 7) убий логин: session + account
      await tx.delete(session).where(eq(session.userId, userId));
      await tx.delete(account).where(eq(account.userId, userId));
    });

    // 8) best-effort Polar анонимизация СЛЕД commit — никога не хвърля
    if (hasPolar) {
      void polarClient!
        .customers.deleteExternal({ externalId: userId, anonymize: true })
        .catch((e) => console.error("Polar customer erase failed", e));
    }
  }
}
```

```ts
import { TRPCError } from "@trpc/server";
import { protectedProcedure } from "@/trpc/init";

// ponytail: in-memory, per-инстанция; при multi-instance deploy → споделен store (Redis). Ceiling приет за M3.3.
const hits = new Map<string, number[]>();

export function checkRateLimit(action: string, userId: string, limit: number, windowMs: number): void {
  const key = `${action}:${userId}`;
  const now = Date.now();
  const arr = (hits.get(key) ?? []).filter((t) => now - t < windowMs);
  if (arr.length >= limit) throw new TRPCError({ code: "TOO_MANY_REQUESTS", message: "TOO_MANY_REQUESTS" });
  arr.push(now);
  hits.set(key, arr);
}

export function rateLimited(action: string, limit: number, windowMs: number) {
  return protectedProcedure.use(({ ctx, next }) => {
    checkRateLimit(action, ctx.user.id, limit, windowMs);
    return next({ ctx: { user: ctx.user } });
  });
}
```
