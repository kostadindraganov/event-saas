# Календарният sync е еднопосочен iCal feed, не Google Calendar OAuth

Вендорът синхронизира потвърдените си резервации към външния си календар чрез подписан, само-за-четене `.ics` feed (`/api/calendar/vendor/<token>.ics`), който добавя като subscription в Google/Apple/Outlook. Платформата НЕ се интегрира с Google Calendar API, не иска OAuth, не съхранява токени на трети страни и не чете външния календар на вендора.

Защо: реалната нужда е «потвърдените ми резервации да се появяват в календара ми». iCal feed я покрива с драстично по-малко повърхност — без OAuth flow, без съхранение/refresh на refresh-токени, без Google verification за scopes, без Google-specific failure modes — и работи с ВСЕКИ календар (Google/Apple/Outlook), не само Google. Компромисите са приети: еднопосочен (само платформа→календар, без внасяне на външна заетост за блокиране на слотове) и polled (календарният клиент опреснява на няколко часа, не instant) — резервациите не са минутно-критични. Feed-ът е bearer-token URL, затова умишлено НЕ съдържа телефон (виж CONTEXT.md «Календарен feed»); токенът е regenerate-ваем при теч.

Upgrade path (ако някога дотрябва real-time или two-way «блокирай ме когато съм зает в Google»): Google Calendar API + OAuth като отделна интеграция, без да чупи feed-а.

Решено в grilling сесия на 2026-07-09 (Фаза 4).
