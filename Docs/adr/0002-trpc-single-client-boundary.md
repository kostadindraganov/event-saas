# tRPC е единствената клиентска граница; RSC чете DAL директно; без Server Actions

Всяка клиентска интеракция (мутации и клиентски заявки — филтри, пагинация, чат) минава през tRPC процедури (auth context → policy → DAL). Публичните SEO страници (обява, категория, търсене) са RSC, които викат DAL-а директно — без слой отгоре. Server Actions не се използват изобщо, въпреки че referentното repo (`demo/pro-dal-local-main`) ги демонстрира — две паралелни мутационни граници означават двойна валидация и объркване къде живее какво.

Следствия: `*.dal.ts` е единственото място, което пипа Drizzle; Zod схемите от `*.dto.ts` се преизползват като `.input()` на tRPC процедурите (валидация на едно място); tRPC v11 RSC prefetch/hydrate pattern (`createTRPCOptionsProxy` + `HydrateClient`) покрива интерактивните страници.
